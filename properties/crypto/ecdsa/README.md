# ECDSA Verification Properties

## Property Summary

| File | Property ID | Description | Compliance |
|------|-------------|-------------|------------|
| `ecdsa_functional.sv` | FG_ECDSA_FUNC_001 | Sign-verify round-trip correctness | PCI-DSS 3.5.1 |
| `ecdsa_nonce.sv` | FG_ECDSA_NONCE_001 | Nonce never repeats across signing operations | Critical |
| `ecdsa_nonce.sv` | FG_ECDSA_NONCE_002 | Nonce generated by approved DRBG | FIPS 186-5 |

## Why Nonce Security Matters

ECDSA nonce reuse is **catastrophic**. If the same nonce `k` is used to sign two different messages, the private key can be trivially recovered:

```
d = (s1 * hash2 - s2 * hash1) * inverse(s1 * k - s2 * k) mod n
```

Real-world exploits: PlayStation 3 key extraction (2010), Bitcoin wallet thefts, Android SecureRandom vulnerability (2013).

FG_ECDSA_NONCE_001 is arguably the single most important property in the ECDSA suite.

## Interface Requirements

Binds through `fg_ecdsa_if` (defined in `interfaces/fg_ecdsa_if.sv`).

Key signals:
- `nonce_k` — The nonce used in signing (must be observable)
- `prev_nonce`, `prev_nonce_valid` — Previous nonce for reuse detection
- `nonce_from_drbg` — Flag indicating approved RNG source
